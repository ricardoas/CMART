Experiment Repeatability

C-MART experiments are logged for repeatability so management algorithms can be directly compared. Clients’ actions and think times from experimental runs are output to XML files which can be read by the workload generator and reproduced in subsequent runs.

There are two types of XML documents used for the repeatability experiments. The first is clientGenerator.xml which describes when each client is added to the system. The second file type is clientn.xml, where n is the index of the client. This file details the sequence of requests that client n will perform.

An example of clientGenerator.xml is as follows:

The HTML4 element at the top indicates the operation mode for the benchmark run. This is followed by a series of client objects. Each object contains the clientIndex number, which is used to refer to the appropriate clientn.xml file. This is followed by the runTime which is the number of milliseconds after the start of the run in which to start the client. Each client contains a clientInfo object. This object contains the client's userID, username, password, and whether or not the client is already registered when first visiting the website. The clientInfo also contains an imageCache, JavaScript/CSS Cache, and an HTML5 cache. These contain files that the client already has in their cache when first visiting the website, and includes SQLite data for the HTML5 Cache.

The clientn.xml document contains details on the clients actions and interactions with the C-MART website. The top of the document contains data on the client's userID, username, and password. This is followed by the actions which the client takes. Each action is indexed by an id number which lets the workload generator which action to take. Each action contains the currentPage reference type, the nextPage action, and the required think time before the action is to be done. Each action contains data on the request. This includes whether it was a GET or POST request and the url of the request. If it is a POST request, then the request object will also contain the data that was sent in the POST.

Due to concurrency issues at the C-MART server it may be impossible to exactly duplicate a run. For example, consider the situation where one client attempts to bid on an item and another client attempts to buy the same item, and the requests are submitted simultaneously. If the bid is processed before the buy, then the first client successfully bids on the item and then the second client successfully buys the item. However if the buy is processed before the bid then the second client successfully buys the item and the first client receives an error message saying that he is attempting to bid on an item that is no longer up for auction. Each of these actions will result in different subsequent actions for each client. If the bid-then-buy order is processed in the original benchmark experiment but the buy-then-bid order is processed in the repeated experiment, then the experiment is no longer repeatable. We provide a statistic for how many clients are unable to repeat their original sequence of actions.

To try and avoid some of the concurrency issues we allow clients to slightly modify the data in their allowable actions. A client may modify the bid price for example if the original bid price is no longer valid. Similarly, the client may want to access an item from a Browse Page that is no longer available. In this situation we allow the user to open a different item. By doing this the repeated experiment stays as close to the original as possible. However the next action (page request type) must always remain the same and the client will exit with an error from the repeatability experiment if this is not possible.
